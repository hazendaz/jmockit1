13  /**
14   * This class makes use of several idioms which would prevent unit testing with more "conventional" mocking tools. Its
15   * usage is as simple as it gets: <code>new MyBusinessService(data).doBusinessOperationXyz()</code>. No need to make
16   * such classes stateless, or worse, <em>singletons</em>; instead, it's designed as a proper object.
17   * <p>
18   * One of those "untestable" idioms is the use of a <em>static persistence facade</em> (the
19   * {@linkplain tutorial.persistence.Database Database} class) for high-level database operations in the context of a
20   * thread-bound work unit. Since all interaction with the facade is through <code>static</code> methods, client classes
21   * cannot be unit tested with a tool which only supports <em>mock objects</em>. With JMockit, though, writing such a
22   * test is just as easy as any other (even easier, in fact, given that <code>static</code> methods don't require an
23   * instance of the mocked class at all).
24   * <p>
25   * Another idiom which runs against limitations of other mocking tools is the direct instantiation and use of external
26   * dependencies, such as the <a href="http://commons.apache.org/email">Apache Commons Email</a> API, used here to send
27   * notification e-mails. As demonstrated here, sending an e-mail is simply a matter of instantiating the appropriate
28   * <code>Email</code> subclass, setting the necessary properties, and calling the <code>send()</code> method. It is
29   * certainly not a good use case for <em>Dependency Injection</em> (DI).
30   * <p>
31   * Finally, consider that application-specific classes like this one are inherently non-reusable in different
32   * contexts/applications; as such, they can and should be made <code>final</code> to reflect the fact that they are not
33   * supposed to be extended through inheritance. In the case of reusable <em>base</em> classes, which are specifically
34   * designed to be extended through inheritance, the judicious use of <code>final</code> for <code>public</code> and
35   * <code>protected</code> methods is important. (The description of the <em>Template Method</em> pattern in the "GoF"
36   * book explains why a properly designed template method should be non-overridable.) Unfortunately, the practice of
37   * <em>designing for extension</em> conflicts with the particular implementation approach employed by other mocking
38   * tools, which dynamically generate a subclass overriding all non-<code>final</code> methods in the mocked class in
39   * order to provide mocked behavior. For JMockit, on the other hand, whether a method or class to be mocked is
40   * <code>final</code> or not is irrelevant, as a radically different mocking approach is employed: class
41   * <em>redefinition</em> as provided by
42   * {@link java.lang.instrument.Instrumentation#redefineClasses(java.lang.instrument.ClassDefinition...)}.
43   */
44  public final class MyBusinessService {
45      private final EntityX data;
46  
47      public MyBusinessService(EntityX data) {
48          this.data = data;
49      }
50  
51      // This method can easily be made transactional, so that any exception thrown during its execution causes a rollback
52      // somewhere up in the call stack (assuming a transaction gets started in the first place).
53      public void doBusinessOperationXyz() throws EmailException {
54          // Locate existing persistent entities of the same entity type (note that the query string is a DSL for querying
55          // persistent domain entities, written in terms of the domain, not in terms of relational tables and columns):
56          List<EntityX> items = find("select item from EntityX item where item.someProperty=?1", data.getSomeProperty());
57  
58          // Compute or obtain from another service a total value for the new persistent entity:
59          BigDecimal total = new BigDecimal("12.30");
60          data.setTotal(total);
61  
62          // Persist the entity (no DAO required for such a common, high-level, operation):
63          persist(data);
64  
65          sendNotificationEmail(items);
66      }
67  
68      private void sendNotificationEmail(List<EntityX> items) throws EmailException {
69          Email email = new SimpleEmail();
70          email.setSubject("Notification about processing of ...");
71          email.addTo(data.getCustomerEmail());
72  
73          // Other e-mail parameters, such as the host name of the mail server, have defaults defined through external
74          // configuration.
75  
76          String message = buildNotificationMessage(items);
77          email.setMsg(message);
78  
79          email.send();
80      }
81  
82      private static String buildNotificationMessage(List<EntityX> items) {
83          StringBuilder message = new StringBuilder();
84  
85          for (EntityX item : items) {
86              message.append(item.getSomeProperty()).append(" Total: ").append(item.getTotal());
87          }
88  
89          return message.toString();
90      }
91  }
